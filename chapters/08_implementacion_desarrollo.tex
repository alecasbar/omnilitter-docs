La fase de desarrollo e implementación de OmniLitter recoge el proceso seguido para transformar los requisitos definidos previamente en un sistema funcional compuesto por una aplicación móvil, un portal web y un backend centralizado. Esta sección describe cómo se organizó el trabajo a lo largo de los distintos sprints, qué partes del sistema se abordaron en cada etapa y qué decisiones técnicas permitieron construir una solución coherente con las necesidades del proyecto. Además, se detallan las funcionalidades implementadas, la integración entre los distintos componentes, el soporte para trabajo offline en la aplicación móvil, la sincronización con el servidor y la estructura general que permite gestionar campañas, sesiones, observaciones, usuarios y grupos dentro de la plataforma.


\clearpage
\section{Sprint 1: Base funcional del sistema}
\subsection{Características y objetivos del sprint}

El primer sprint tuvo como objetivo establecer la base funcional de OmniLitter.
En esta primera iteración se buscó construir una versión mínima pero coherente
del sistema, capaz de soportar los flujos principales sobre los que se apoyaría
el resto del desarrollo. Para ello, se abordaron las funcionalidades esenciales
relacionadas con el acceso de usuarios, la separación de datos por grupos de
trabajo, la creación inicial de campañas y sesiones de muestreo, y el registro
básico de observaciones desde la aplicación móvil.

Este sprint fue especialmente importante porque permitió definir la estructura
operativa inicial del sistema. A partir de esta base, los usuarios podían
acceder a la plataforma, trabajar dentro de un grupo concreto, crear campañas
como contexto organizativo del trabajo de campo y registrar las primeras
observaciones asociadas a una sesión. De este modo, se estableció el flujo
principal de uso de OmniLitter, que posteriormente sería ampliado con
capacidades de geolocalización, fotografías, trabajo offline, sincronización y
explotación de datos.

\begin{itemize}[leftmargin=1.4em]
\item \textbf{Autenticación y acceso.} Esta funcionalidad constituye el punto de entrada al sistema y permite identificar de forma segura a cada usuario antes de acceder a las zonas privadas. Abarca el registro, el inicio y cierre de sesión, la conservación de la sesión válida y la protección de rutas internas tanto en el portal web como en la aplicación móvil.
\item \textbf{Usuarios, grupos y permisos.} Esta parte funcional organiza el trabajo colaborativo mediante grupos, membresías y roles. Su finalidad es delimitar qué datos puede consultar o modificar cada usuario, qué grupo actúa como contexto activo de trabajo y cómo se separa la información entre equipos diferentes.
\item \textbf{Campañas.} La campaña funciona como unidad organizativa del trabajo de campo. Sirve para planificar una actividad de muestreo dentro de un grupo, agrupar las sesiones que pertenecen al mismo contexto y mantener una referencia común para consultar después las observaciones registradas.
\item \textbf{Sesiones de muestreo.} La sesión representa una salida o actividad concreta de recogida de datos dentro de una campaña. Esta funcionalidad permite relacionar el trabajo realizado con un grupo, un observador, una campaña y el conjunto de observaciones generadas durante esa actividad.
\item \textbf{Observaciones.} La observación es el registro básico de un residuo encontrado durante una sesión. Esta funcionalidad recoge la información necesaria para describirlo, como taxonomía, coordenadas, cantidades, materiales, categorías y notas, manteniendo además su relación con la sesión, la campaña, el grupo y el usuario responsable.
\end{itemize}

\subsection{Implementación realizada}

El primer sprint se dedicó a construir la base técnica sobre la que se apoyaría
el resto de OmniLitter. La implementación se centró en disponer de una
arquitectura operativa mínima, con autenticación, separación de datos por
grupos, gestión inicial de campañas y sesiones, y registro básico de
observaciones desde la aplicación móvil.

\paragraph{Backend y modelo de dominio.}
Se estructuró una API REST modular sobre Express \cite{express}, con módulos diferenciados
para autenticación, usuarios, grupos, campañas, sesiones, observaciones y
taxonomía. Las rutas privadas quedaron protegidas mediante middleware de
autenticación JWT \cite{rfc7519jwt} y las reglas de acceso se articularon alrededor del grupo
activo del usuario. Esta decisión permitió separar el rol global del usuario
del rol que ejerce dentro de cada grupo, distinguiendo entre administración
general, administración de grupo e investigador de campo.

En autenticación se implementaron registro, inicio de sesión, cierre de sesión,
refresco de sesión y recuperación de contraseña. Las contraseñas se almacenaron
mediante hash y los tokens de refresco quedaron persistidos para poder revocar
sesiones. En el modelo de datos consolidado se definieron las entidades
principales del sistema: usuarios, tokens, grupos, membresías, invitaciones,
campañas, sesiones de muestreo, taxonomía, observaciones e ítems de residuo.
No se localizaron migraciones incrementales por sprint; el repositorio conserva
una migración inicial con el estado final consolidado.

El flujo de sesión se diseñó con separación entre token de acceso y token de
refresco. El access token contiene únicamente el identificador del usuario y su
rol global, mientras que el refresh token se almacena en base de datos como hash
y se rota en cada refresco. De este modo, el cierre de sesión, el cambio de
contraseña o la recuperación de contraseña pueden revocar sesiones existentes
sin depender solo de la expiración del JWT. La recuperación de contraseña se
implementó con tokens aleatorios de un solo uso, almacenados como hash
\texttt{SHA-256} y con caducidad de 30 minutos.

La autorización por grupos se resolvió con funciones de dominio reutilizadas por
campañas, sesiones, observaciones, métricas y administración. El backend
distingue entre poder acceder a un grupo, poder administrarlo y resolver el
alcance completo de grupos visibles para el usuario autenticado. Esta capa es
la que permite que un superadministrador vea información agregada, que un
administrador gestione su grupo y que un investigador quede limitado a sus
propios datos cuando corresponde.

\paragraph{Portal web.}
El portal web se organizó como una aplicación React \cite{react} con rutas protegidas,
conservación de sesión en el navegador y un layout común para las áreas
privadas. Sobre esta base se integraron las primeras pantallas de campañas,
grupos y administración de usuarios. La navegación interna quedó preparada para
incorporar progresivamente módulos posteriores como observaciones, mapa,
analítica y configuración.

\paragraph{Aplicación móvil y paquete compartido.}
La aplicación móvil se implementó con Expo y React Native \cite{expo,reactnative}. Se definió el flujo
de autenticación, la navegación principal y los primeros servicios de consumo
de la API. Desde la sección de campañas se incorporó la creación inicial de
sesiones y observaciones, utilizando contratos compartidos para evitar
divergencias entre clientes y servidor. El paquete común centralizó tipos y
esquemas Zod \cite{zod} de autenticación, campañas, sesiones, observaciones, ítems y
fotografías, de modo que la validación de entrada no dependiera únicamente de
cada interfaz.

\paragraph{Integración entre capas.}
La integración se resolvió mediante servicios REST consumidos por web y móvil,
tokens JWT en las peticiones privadas y validación compartida en
\texttt{packages/shared}. Las campañas y sesiones quedaron vinculadas al grupo
activo, mientras que las observaciones se asociaron a una sesión, una taxonomía
y un usuario observador. Con ello se estableció el flujo técnico básico:
autenticación, selección de contexto de trabajo, planificación de campaña,
creación de sesión y registro inicial de residuos.

\subsection{Resultado del sprint}

El resultado del primer sprint fue la obtención de una primera versión funcional de OmniLitter, concebida como una base mínima pero operativa sobre la que construir el resto del sistema. Esta versión permitió comprobar que la arquitectura planteada para el Trabajo de Fin de Grado era técnicamente viable, al integrar de forma coordinada los principales bloques del proyecto: backend, portal web, aplicación móvil y paquete compartido de validaciones y tipos comunes.

Al finalizar la iteración, el sistema ya disponía de los elementos esenciales para representar el flujo principal de trabajo previsto en OmniLitter. En concreto, se implementó una primera base de autenticación de usuarios, la organización de la información mediante grupos de trabajo, la gestión inicial de campañas y sesiones de muestreo, así como el registro básico de observaciones desde la aplicación móvil. Aunque estas funcionalidades no constituían todavía una versión completa del producto final, sí permitían recorrer de forma coherente el flujo fundamental del sistema: acceso del usuario, pertenencia a un grupo, creación o selección de una campaña, apertura de una sesión de muestreo y registro de observaciones asociadas.

Desde el punto de vista técnico, este sprint tuvo especial importancia porque permitió validar las decisiones iniciales de arquitectura. La separación entre plataformas, el uso de una API como punto central de comunicación, la existencia de un paquete compartido entre aplicaciones y la persistencia de datos en una base de datos relacional demostraron ser adecuados para soportar el crecimiento posterior del proyecto. Además, la integración entre la aplicación móvil y el backend permitió comprobar que el sistema podía evolucionar hacia escenarios más avanzados, como la captura de datos en campo, la gestión de evidencias fotográficas, la persistencia local y la sincronización diferida.

Desde el punto de vista funcional, el sprint consolidó la estructura básica del dominio de OmniLitter. Durante su desarrollo se identificó la necesidad de introducir de forma explícita el concepto de campaña dentro del flujo de trabajo, ya que este elemento resultaba imprescindible para organizar correctamente las sesiones y las observaciones. Esta adaptación supuso un reajuste respecto a la planificación inicial, pero permitió obtener un modelo más coherente con las necesidades reales del sistema y con la forma en que se espera que los usuarios organicen las actividades de muestreo.

Asimismo, la iteración permitió detectar y resolver decisiones relevantes relacionadas con la gestión de permisos. Inicialmente, el control de acceso podía plantearse únicamente a partir de roles generales, pero el desarrollo del sprint puso de manifiesto que era necesario considerar también la pertenencia de cada usuario a un grupo concreto y su rol dentro de ese grupo. Esta decisión resultó especialmente importante para garantizar que la información quedase correctamente segmentada y que cada usuario solo pudiera acceder a los datos correspondientes a su contexto de trabajo.

En términos académicos, el primer sprint aportó una primera evidencia clara de trazabilidad entre los requisitos definidos, el diseño del sistema y la implementación realizada. Las funcionalidades desarrolladas materializaron los requisitos iniciales relacionados con autenticación, organización por grupos, campañas, sesiones y observaciones. A su vez, las incidencias surgidas durante la iteración reflejaron un proceso normal de refinamiento del dominio, propio de un desarrollo incremental, en el que algunas decisiones se ajustan conforme se contrastan con la implementación real.

El grado de cumplimiento del sprint se considera adecuado. Aunque fue necesario introducir ciertos ajustes sobre la planificación inicial, estos no comprometieron el objetivo principal de la iteración. Por el contrario, contribuyeron a mejorar la consistencia del sistema y a dejar una base más sólida para los siguientes ciclos de desarrollo. Al cierre del sprint, OmniLitter contaba con una estructura funcional suficiente para demostrar la viabilidad del proyecto y para continuar incorporando funcionalidades de mayor complejidad en las iteraciones posteriores.

En consecuencia, este primer sprint cumplió su papel como punto de partida del desarrollo incremental. Su principal aportación no fue únicamente la entrega de una serie de funcionalidades iniciales, sino la validación de la arquitectura, del modelo de datos básico y del flujo principal de uso. Gracias a ello, los siguientes sprints pudieron centrarse en ampliar la captura de información en campo, mejorar la experiencia de usuario, incorporar persistencia local, reforzar la gestión de permisos y avanzar hacia una versión más completa y próxima a los objetivos finales del Trabajo de Fin de Grado.

\section{Sprint 2: Captura de campo y análisis inicial}
\subsection{Características y objetivos del sprint}

El segundo sprint se centró en ampliar el flujo de trabajo de campo y mejorar
la calidad de los datos registrados desde la aplicación móvil. Una vez
construida la base funcional del sistema, esta iteración permitió enriquecer las
sesiones y observaciones con información más precisa, incorporando coordenadas,
fotografías, metadatos, validaciones y mecanismos de revisión previa.

Además, este sprint introdujo las primeras funcionalidades de explotación de
datos desde el portal web. Esto permitió que la información capturada desde la
aplicación móvil no quedara limitada al registro, sino que pudiera consultarse,
filtrarse e interpretarse desde la parte web del sistema. Por tanto, este sprint
supuso un avance tanto en la calidad de la captura en campo como en la capacidad
inicial de análisis de la información registrada.

\begin{itemize}[leftmargin=1.4em]
\item \textbf{Sesiones de muestreo II.} Esta funcionalidad continúa el trabajo iniciado en las sesiones de muestreo, ampliándolas para que representen una actividad de campo con información espacial y de progreso. Además de identificar la campaña y el observador, la sesión pasa a reflejar el recorrido realizado, la distancia aproximada y el avance del muestreo, datos necesarios para interpretar posteriormente la cobertura del trabajo.
\item \textbf{Observaciones II.} Esta parte funcional continúa la funcionalidad de observaciones, ampliando el registro básico de residuos con coordenadas, fotografías y metadatos asociados. También contempla la revisión y corrección de datos antes de su consolidación, así como reglas de validación y permisos para mantener la coherencia científica y el control de acceso.
\item \textbf{Trabajo offline.} Esta funcionalidad permite que la aplicación móvil siga siendo útil cuando la conectividad es limitada. Su objetivo es conservar localmente la información necesaria para navegar, consultar datos ya disponibles y continuar el trabajo de campo sin depender de una conexión constante.
\item \textbf{Consulta web de observaciones.} Esta funcionalidad convierte el portal web en una herramienta de revisión de los datos capturados. Permite localizar observaciones mediante filtros, abrir su detalle, comprobar la información asociada y consultar fotografías o metadatos relevantes desde un entorno más cómodo para la supervisión.
\item \textbf{Dashboard y métricas.} El dashboard ofrece una visión agregada de la actividad del grupo activo. Su función es resumir el estado general del trabajo mediante totales, distribuciones e indicadores sobre campañas, sesiones, observaciones, residuos y fotografías.
\item \textbf{Analítica y exportación.} Esta parte funcional está orientada a explorar los datos registrados y preparar su reutilización. Permite trabajar con consultas filtradas, revisar subconjuntos de información y establecer la base para generar salidas exportables en fases posteriores del desarrollo.
\end{itemize}
\subsection{Implementación realizada}

El segundo sprint amplió la base anterior hacia un flujo de campo más completo.
La implementación se centró en enriquecer las sesiones y observaciones con
coordenadas, recorridos, fotografías y persistencia local, además de incorporar
las primeras capacidades web de consulta, métricas y análisis sobre los datos
capturados.

\paragraph{Captura móvil y persistencia local.}
En la aplicación móvil se incorporó la obtención de ubicación mediante
\texttt{expo-location} \cite{expo-location}, con solicitud de permisos y lectura de coordenadas del
dispositivo. Para las sesiones se implementó el registro de recorridos mediante
puntos GPS, aplicando filtros de precisión, distancia mínima y velocidad máxima.
Esta información permitió representar el avance de una salida de campo mediante
\texttt{trackPoints} y \texttt{sampledArea}, mientras que cada observación
incorporó sus propias coordenadas.

Durante una sesión activa se registra el recorrido con
lecturas periódicas cada tres segundos o cada metro de desplazamiento, según
los avisos que proporcione el sistema operativo. Sobre esas lecturas se añadió un filtro específico en el servicio móvil
\texttt{samplingTrack.ts}. Cada punto se acepta solo si contiene coordenadas
válidas, una fecha de registro interpretable, una precisión no superior a 15
metros, una separación mínima de 2,5 metros respecto al punto anterior y una
velocidad implícita no superior a 8 m/s. La distancia entre dos puntos se
calcula mediante la fórmula de Haversine, por ser
suficiente para estimar recorridos cortos sobre coordenadas geográficas sin
introducir una dependencia geoespacial adicional. Estos umbrales no proceden de
una norma externa, sino de una decisión de diseño del proyecto para reducir
saltos espurios del GPS sin perder desplazamientos reales de muestreo.

El backend repite parte de esta limpieza antes de persistir el progreso de la
sesión. En \texttt{apps/api/src/modules/sessions/sessions.service.ts} se
normalizan los puntos, se ordenan temporalmente, se eliminan duplicados y se
vuelven a descartar lecturas no plausibles. En servidor se usa una tolerancia
algo más amplia para no rechazar datos ya capturados en campo: precisión máxima
de 50 metros, separación mínima de 1,5 metros y la misma velocidad máxima de
8 m/s. Además, los esquemas compartidos limitan los recorridos a un máximo de
5000 puntos por actualización y validan latitud, longitud y precisión.

El área muestreada se genera a partir del recorrido filtrado. Para muestreo
libre se construye una envolvente rectangular con un margen de 3 metros; para
transectos se usa como margen la mitad de la anchura del transecto, con 5 metros
como valor por defecto; y para cuadrículas se usa la mitad del tamaño de celda,
con 10 metros como valor por defecto, ajustando después los límites a pasos de
celda. Esta geometría se guarda como \texttt{sampledArea}, mientras que los
puntos crudos aceptados permanecen en \texttt{trackPoints}. Así, el sistema
conserva tanto la trayectoria original como una superficie simplificada para
mapas, métricas y exportaciones.

También se añadió la captura y selección de fotografías desde cámara o galería.
Las imágenes se copiaron al almacenamiento local de la aplicación para poder
mantenerlas asociadas a observaciones incluso en escenarios de conectividad
variable. La base SQLite local \cite{sqlite} se estructuró mediante repositorios por entidad,
incluyendo usuarios, grupos, campañas, sesiones, observaciones, taxonomía,
fotografías y cola de sincronización. Esta separación facilitó que la interfaz
móvil pudiera revisar, editar o eliminar datos creados localmente antes de su
envío al servidor.

\paragraph{Backend y validación científica.}
En el backend se reforzó el módulo de observaciones para aceptar fotografías,
registrar sus metadatos y exponer endpoints de listado, descarga, subida y
eliminación. Además, se añadieron reglas de validación sobre taxonomía activa,
coordenadas, fechas, cantidades y permisos por grupo. La lógica impidió, por
ejemplo, registrar observaciones fuera del intervalo temporal permitido para la
sesión o asociarlas a grupos a los que el usuario no tuviera acceso.

La gestión de fotografías se implementó con endpoints específicos para no
mezclar la edición de campos científicos de la observación con la transferencia
de ficheros. El backend recibe las imágenes codificadas en base64, valida que el
tipo MIME pertenezca a formatos soportados como JPEG, PNG, WebP, HEIC o HEIF,
rechaza imágenes vacías o superiores a 10 MB y limita cada observación a cinco
fotografías. Antes de guardar el archivo, normaliza el nombre recibido, calcula
un \texttt{checksum} \texttt{SHA-256}, lo almacena bajo el directorio de subidas
configurado y registra metadatos como tamaño, tipo MIME, dimensiones, origen,
fecha de captura, EXIF y usuario que subió la imagen. Si falla la persistencia
en base de datos, el archivo físico se elimina para evitar residuos
inconsistentes.

El modelo consolidado incorporó fotografías de observación, URLs asociadas a
observaciones y campos espaciales de sesión. Aunque estos cambios aparecen en
la migración inicial consolidada, su uso funcional corresponde a esta fase de
captura avanzada.

\paragraph{Portal web y explotación inicial.}
El portal web incorporó listados y fichas de observaciones con filtros por
campaña, sesión, entorno, observador, fecha y presencia de fotografías. La ficha
permitió consultar la información científica y contextual del registro,
incluyendo taxonomía, ítems, coordenadas, notas y evidencias fotográficas.

Además, se implementó el cálculo de métricas agregadas en backend y su
visualización en el dashboard web. Estas métricas resumieron actividad,
observaciones e ítems por dimensiones como campaña, sesión, categoría, material,
entorno u observador. El módulo de analítica inicial reutilizó esta información
para explorar subconjuntos filtrados y preparar exportaciones posteriores.

\paragraph{Integración entre capas.}
La aplicación móvil construyó payloads compatibles con los esquemas compartidos
y los conservó localmente hasta poder enviarlos. El backend validó y persistió
observaciones, ítems y fotografías, y el portal web consumió esos datos para
revisión y análisis. De esta forma, el dato capturado en campo comenzó a
alimentar directamente la supervisión web y las primeras métricas del sistema.

\subsection{Resultado del sprint}

El resultado del segundo sprint fue una evolución significativa de OmniLitter desde una versión funcional mínima hacia una herramienta más cercana a un escenario real de trabajo de campo. Mientras que el primer sprint permitió validar la estructura general del sistema y el flujo básico de campañas, sesiones y observaciones, esta segunda iteración se centró en enriquecer la captura de información, mejorar la utilidad científica de los datos registrados y reforzar la conexión entre la aplicación móvil, el backend y el portal web.

Al cierre del sprint, la aplicación móvil ya permitía registrar observaciones con un mayor nivel de detalle, incorporando coordenadas geográficas, fotografías asociadas y metadatos relevantes para contextualizar la recogida de información. Este avance resultó especialmente importante, ya que las observaciones dejaron de ser simples registros textuales o básicos para convertirse en unidades de información más completas, verificables y útiles desde el punto de vista del análisis posterior. La inclusión de datos espaciales y evidencias visuales aumentó la calidad del registro y aproximó el sistema a las necesidades reales de una campaña de monitorización ambiental.

Otro resultado relevante fue la introducción de una primera capacidad de persistencia local en la aplicación móvil. Esta funcionalidad permitió avanzar hacia uno de los objetivos fundamentales del proyecto: garantizar que el trabajo de campo pueda realizarse en contextos donde la conectividad no siempre esté disponible o sea estable. Aunque la sincronización completa se abordaría en iteraciones posteriores, este sprint dejó preparada la base necesaria para almacenar información localmente y reducir la dependencia inmediata de la conexión con el servidor. Desde el punto de vista técnico, este avance supuso un paso importante en la adaptación de OmniLitter a las condiciones reales de uso fuera de un entorno controlado.

En paralelo, el portal web comenzó a consolidarse como una herramienta de supervisión y consulta de los datos recogidos desde la aplicación móvil. La posibilidad de visualizar observaciones, consultar información asociada y disponer de primeras métricas permitió comprobar que los datos generados en campo podían ser transferidos conceptualmente al entorno de análisis y revisión. De este modo, el sistema dejó de funcionar como un conjunto de módulos aislados y empezó a mostrar un ciclo de trabajo más completo: captura de datos desde la aplicación móvil, validación y almacenamiento mediante el backend, y consulta posterior desde el portal web.

Desde una perspectiva académica, este sprint aportó una evidencia clara de integración funcional entre las distintas capas del sistema. No se trató únicamente de añadir nuevas pantallas o servicios independientes, sino de ampliar el flujo principal del proyecto con información más rica y con mayor valor para el usuario final. La iteración permitió demostrar que la arquitectura definida en el sprint anterior era capaz de soportar funcionalidades más complejas, como la gestión de imágenes, el tratamiento de coordenadas, el almacenamiento local y la explotación inicial de datos desde el portal web.

Durante el desarrollo del sprint también se identificaron algunas dificultades técnicas relevantes, especialmente relacionadas con la configuración de SQLite en la aplicación móvil, la gestión de ficheros locales y las pruebas en dispositivos o emuladores. Estas incidencias ralentizaron parte del proceso de validación, pero no impidieron alcanzar los objetivos principales de la iteración. Por el contrario, permitieron detectar de forma temprana limitaciones propias del entorno móvil y ajustar la implementación para que el sistema resultante fuera más robusto de cara a los siguientes sprints.

El grado de cumplimiento del sprint se considera adecuado, ya que se alcanzaron los avances funcionales previstos y se reforzó de forma clara el valor práctico de OmniLitter. La aplicación móvil incorporó capacidades más representativas del trabajo de campo, el backend amplió su papel como capa de validación y almacenamiento, y el portal web comenzó a ofrecer una visión inicial de los datos registrados. Todo ello contribuyó a consolidar el sistema como una plataforma integrada, en la que las distintas partes del proyecto cumplen funciones complementarias dentro de un mismo flujo de trabajo.

En consecuencia, el segundo sprint supuso un punto de maduración importante dentro del desarrollo incremental. Su principal aportación fue transformar la base funcional inicial en un sistema más orientado al uso real, incorporando información geográfica, evidencias fotográficas, metadatos y persistencia local. Además, dejó preparado el proyecto para abordar en las siguientes iteraciones aspectos clave como la sincronización completa entre dispositivo y servidor, la representación espacial de las observaciones, la mejora de la experiencia de usuario y la consolidación del ciclo completo de captura, consulta y análisis de datos ambientales.

\section{Sprint 3: Sincronización, mapas y exportación científica}
\subsection{Características y objetivos del sprint}

El tercer sprint tuvo como objetivo completar el ciclo de trabajo entre la
aplicación móvil y el servidor, especialmente en escenarios donde la
conectividad no estuviera garantizada. En esta iteración se abordó la

conservación de datos locales hasta su envío y la gestión de resultados
individuales dentro de los procesos de sincronización.

Junto a ello, se incorporaron funcionalidades de visualización geográfica y
configuración del usuario. Esto permitió que OmniLitter pasase de ser una
herramienta centrada en la captura y consulta básica de datos a ofrecer un flujo
más completo de trabajo: registro en campo, persistencia local, sincronización
con el backend, representación espacial de la información y adaptación de
ciertas preferencias a las necesidades del usuario.

\begin{itemize}[leftmargin=1.4em]
\item \textbf{Trabajo offline y sincronización II.} Esta funcionalidad continúa el trabajo offline desarrollado previamente, ampliándolo con la sincronización entre la aplicación móvil y el backend cuando vuelve a existir conectividad. Gestiona los cambios pendientes, agrupa operaciones, conserva los datos hasta confirmar su envío y permite que sesiones, observaciones y progresos capturados en campo acaben integrados en el sistema central.
\item \textbf{Sesiones de muestreo III.} Esta parte funcional continúa la evolución de las sesiones de muestreo, ampliándolas para que puedan modificarse localmente y sincronizarse después con el servidor. Incluye la creación, actualización del progreso y finalización de sesiones, además del uso de rutas y áreas muestreadas como información útil para mapas, análisis y exportaciones.
\item \textbf{Observaciones III.} Esta funcionalidad continúa la ampliación de las observaciones, centrándose ahora en el paso de los registros creados en el dispositivo al backend. Su objetivo es que los datos capturados sin conexión, junto con sus elementos asociados, queden publicados posteriormente y cada operación de sincronización pueda evaluarse de forma individual.
\item \textbf{Visualización geográfica.} Esta funcionalidad permite interpretar los datos desde una perspectiva espacial. El mapa representa observaciones, rutas y áreas muestreadas, facilitando la identificación de zonas ya recorridas, puntos de registro y posibles concentraciones de residuos.
\item \textbf{Analítica y exportación II.} Esta parte funcional continúa la base de analítica y exportación del sprint anterior, ampliándola hacia la reutilización externa de los datos de OmniLitter. Las exportaciones en CSV y GeoJSON permiten trasladar observaciones, resúmenes o información geográfica a herramientas de análisis científico, hojas de cálculo o sistemas cartográficos.
\item \textbf{Perfil y configuración.} Esta funcionalidad agrupa las opciones que permiten adaptar la aplicación al usuario y al contexto de uso. Incluye datos de perfil, cambio de contraseña, selección de idioma y preferencias de sincronización, de forma que el comportamiento del sistema pueda ajustarse a las condiciones del dispositivo y de la conexión.
\end{itemize}
\subsection{Implementación realizada}

El tercer sprint se orientó a cerrar el ciclo técnico entre la aplicación móvil,
el backend y el portal web. La implementación se centró en sincronización,
visualización geográfica, exportación de datos y configuración de preferencias,
permitiendo que los registros creados sin conexión pudieran consolidarse y
explotarse posteriormente desde el sistema central.

\paragraph{Sincronización móvil-backend.}
En la aplicación móvil se implementó una pantalla específica de sincronización
y un servicio encargado de procesar operaciones pendientes. La base local
conservó las acciones en una cola persistente, de forma que sesiones,
actualizaciones de progreso, finalizaciones y observaciones pudieran mantenerse
hasta recuperar conectividad. También se añadió sincronización automática al
volver a estar online, considerando preferencias como la restricción a redes
Wi-Fi.

El backend incorporó endpoints de sincronización por lotes y consulta de cambios
remotos. El procesamiento por lotes permitió enviar varias operaciones en una
misma petición y obtener resultados individuales por elemento, evitando que un
fallo puntual invalidase necesariamente el conjunto completo. Esta estructura
resultó especialmente adecuada para escenarios de campo, donde los datos pueden
acumularse en el dispositivo durante periodos sin conexión.

La sincronización no se planteó como una subida genérica de tablas, sino como
una cola de operaciones de captura con tipos explícitos. El contrato compartido
define operaciones diferenciadas para crear sesiones, editar sus campos,
actualizar su progreso, finalizarlas y crear observaciones, agrupadas en lotes
de hasta 500 elementos. La app móvil genera identificadores de cliente para
relacionar cada operación enviada con la respuesta del servidor; si una
operación se acepta, se marca como sincronizada o se elimina de
\texttt{sync\_outbox}, y si falla se incrementa el contador de intentos y se
conserva el último error. Para evitar sobrescribir datos locales pendientes, la
descarga de cambios remotos se omite cuando existe trabajo sin sincronizar en
el dispositivo.

\paragraph{Mapa y visualización espacial.}
El portal web incorporó un mapa interactivo basado en React Leaflet \cite{reactleaflet}. Se
implementaron marcadores de observaciones, agrupación visual, filtros por
campaña, entorno, categoría y fecha, y carga de fotografías en ventanas de
detalle. Además, se añadieron modos de visualización de densidad, recorridos y
áreas muestreadas, utilizando los datos espaciales de sesiones y observaciones
persistidos por el backend.

Para las sesiones de muestreo, la API expuso rutas y áreas asociadas al grupo
activo. El portal pudo así representar recorridos, superficies muestreadas y
zonas de concentración de residuos, conectando la captura móvil con una lectura
geográfica de los resultados.

En la exportación geográfica, el mapa construye una
\texttt{FeatureCollection} GeoJSON con coordenadas en el orden
\texttt{[longitud, latitud]}. Cuando el usuario exporta desde el modo de rutas,
se incluyen primero polígonos de área muestreada, después líneas de recorrido y
por último puntos de observación, de forma que los visores GIS puedan interpretar
por separado superficies, trayectorias y registros puntuales. Las propiedades de
cada elemento incorporan contexto de campaña, sesión, entorno, tipología,
observador, método de muestreo y distancia cuando procede.

\paragraph{Analítica, exportaciones y configuración.}
Se ampliaron las utilidades de exportación del portal web con salidas CSV, JSON
y GeoJSON. El dashboard pudo exportar datos agregados, el mapa generó
colecciones geográficas de observaciones, rutas y áreas, y la analítica produjo
ficheros orientados al tratamiento posterior de observaciones, residuos y
resúmenes. En backend, las consultas de analítica incluyeron información de
campaña, grupo, sesión, entorno, observador, coordenadas, taxonomía, método de
muestreo, duración e ítems.

Las exportaciones CSV se generaron sin depender de librerías externas. El
servicio web añade marca BOM UTF-8 para conservar acentos al abrir los ficheros
en hojas de cálculo, utiliza punto y coma como separador habitual en entornos
españoles y escapa comillas, saltos de línea y separadores. La preferencia de
formato de exportación se conserva en \texttt{localStorage}, por lo que el
usuario puede mantener CSV o JSON como formato por defecto entre sesiones.

En la aplicación móvil se incorporaron preferencias de idioma, sincronización y
precisión de ubicación. La infraestructura de internacionalización también se
observa en el portal web, aunque parte de la configuración web se completó en el
sprint siguiente según la trazabilidad disponible.

\paragraph{Integración entre capas.}
La app móvil trabajó sobre SQLite, agrupó cambios pendientes y los envió al
backend cuando la conectividad lo permitió. La API persistió los cambios en la
base central y expuso tanto cambios remotos como datos espaciales y analíticos.
El portal web consumió esos datos ya consolidados para mapas, métricas y
exportaciones. La integración de este sprint unió captura offline, publicación,
consulta y reutilización externa de los datos.

\subsection{Resultado del sprint}

El resultado del tercer sprint fue un avance especialmente relevante en la madurez funcional de OmniLitter, ya que permitió cerrar gran parte del ciclo de trabajo previsto para el sistema. Si en las iteraciones anteriores se había construido la base funcional y se había enriquecido la captura de información en campo, este sprint consolidó la conexión entre el uso offline de la aplicación móvil, la sincronización con el backend, la consulta desde el portal web, la visualización geográfica y la exportación de datos para su posterior reutilización.

Al finalizar la iteración, OmniLitter ya permitía conservar información en el dispositivo móvil cuando no existía conectividad, sincronizar los datos con el servidor una vez recuperada la conexión y poner esa información a disposición del portal web. Este resultado supuso un salto cualitativo respecto a los sprints anteriores, ya que el sistema dejó de depender de un uso conectado en todo momento y comenzó a aproximarse a las condiciones reales de una campaña de monitorización ambiental, donde el trabajo de campo puede desarrollarse en zonas con cobertura limitada o inestable.

Desde el punto de vista funcional, el sprint permitió completar un recorrido de uso mucho más representativo del producto final. El usuario podía registrar sesiones y observaciones desde la aplicación móvil, mantener dichos datos almacenados localmente, sincronizarlos posteriormente con el backend y consultarlos desde el portal web. A partir de esta información, el sistema ofrecía mecanismos iniciales de interpretación mediante listados, métricas, mapas y opciones de exportación. De este modo, OmniLitter empezó a comportarse como una plataforma integrada y no únicamente como un conjunto de funcionalidades independientes.

Uno de los resultados más importantes de esta iteración fue la consolidación de la sincronización como elemento central del sistema. Esta funcionalidad permitió conectar de forma efectiva el contexto de captura de datos en campo con el entorno de análisis y supervisión. La aplicación móvil asumió el papel de herramienta de recogida de información, el backend actuó como punto de validación y consolidación de los datos, y el portal web se convirtió en el espacio desde el que revisar, representar y reutilizar la información obtenida. Esta separación de responsabilidades reforzó la coherencia de la arquitectura planteada inicialmente.

La incorporación de la visualización geográfica también aportó un valor significativo al proyecto. La representación de sesiones y observaciones sobre mapas permitió contextualizar espacialmente los datos registrados y comprobar que la información georreferenciada capturada desde la aplicación móvil podía ser utilizada posteriormente desde el portal web. En un sistema orientado a la monitorización ambiental, esta capacidad resulta especialmente relevante, ya que la ubicación de los residuos y la delimitación de las zonas de muestreo son elementos fundamentales para interpretar correctamente los datos.

Asimismo, la exportación de información supuso otro avance importante en la utilidad práctica de OmniLitter. Esta funcionalidad permitió plantear el sistema no solo como una herramienta de registro y consulta interna, sino también como una fuente de datos reutilizables en otros entornos de análisis. Desde una perspectiva académica, este aspecto resulta especialmente valioso, ya que conecta la implementación realizada con uno de los objetivos habituales en proyectos de carácter científico: facilitar la explotación posterior de los datos recogidos.

Durante el desarrollo del sprint se produjeron algunas desviaciones relacionadas con la coordinación entre los datos locales y remotos, la coherencia visual de determinadas pantallas y la validación completa del flujo de sincronización. Estas desviaciones se gestionaron mediante ajustes de planificación y refinamientos técnicos, priorizando en todo momento la estabilidad del recorrido funcional principal. Aunque la sincronización se validó mediante pruebas manuales completas y recorridos funcionales de extremo a extremo, se dejó documentada como limitación la ausencia de una suite automatizada específica para este proceso.

El grado de cumplimiento del sprint se considera alto, ya que se alcanzó uno de los hitos más importantes del desarrollo incremental: demostrar que los datos podían capturarse en campo, persistirse localmente, sincronizarse con el servidor y explotarse posteriormente desde el portal web. Este resultado aportó una evidencia clara de integración entre los tres componentes principales del proyecto y permitió validar que la arquitectura definida era capaz de soportar el flujo completo de uso previsto para OmniLitter.

En consecuencia, el tercer sprint representó un punto de consolidación dentro del desarrollo del Trabajo de Fin de Grado. Su principal aportación fue transformar las funcionalidades construidas en iteraciones anteriores en un flujo integrado y coherente, capaz de conectar la recogida de información en campo con su posterior análisis y reutilización. Gracias a este avance, el proyecto llegó al sprint final con las funcionalidades centrales ya implementadas, quedando pendiente principalmente el refinamiento de la experiencia de usuario, la validación final, la preparación del despliegue y el cierre documental de la solución desarrollada.

\subsection{Características y objetivos del sprint}

El cuarto sprint tuvo un carácter principalmente orientado a la consolidación
del producto. Una vez implementados los principales flujos funcionales, esta
iteración se centró en reforzar funcionalidades ya existentes, cerrar aspectos
pendientes detectados durante el desarrollo, mejorar la experiencia de usuario y
asegurar una versión más estable y presentable para la validación funcional del
Trabajo de Fin de Grado.

En esta etapa se revisaron tanto la administración web como el modelo científico
que condiciona campañas, sesiones y observaciones. También se realizaron
ajustes de interfaz, limpieza de elementos no utilizados y refinamientos en las
validaciones. El objetivo final fue reducir inconsistencias, mejorar la
coherencia general del sistema y preparar una versión suficientemente completa
para su evaluación.

\begin{itemize}[leftmargin=1.4em]
\item \textbf{Configuración y administración II.} Esta funcionalidad continúa la organización de usuarios, grupos y permisos, ampliándola con ajustes administrativos necesarios para operar la plataforma con coherencia desde el portal web. Abarca la gestión de usuarios, grupos, membresías y permisos, asegurando que cada rol actúe dentro del alcance que le corresponde.
\item \textbf{Campañas II.} Esta parte funcional continúa la funcionalidad de campañas, reforzando su papel organizativo dentro del grupo y su relación con las sesiones asociadas. La clasificación científica fina no queda fijada en la campaña, sino en cada sesión de muestreo, lo que permite que una misma campaña agrupe salidas realizadas en entornos o tipologías distintas.
\item \textbf{Sesiones de muestreo IV.} Esta funcionalidad continúa la evolución de las sesiones de muestreo, ampliándolas como registros editables y validados científicamente. Permite ajustar datos de la salida de campo, asociar el entorno correspondiente y aplicar reglas compartidas con el catálogo para mantener coherencia entre la campaña planificada y el muestreo realizado.
\item \textbf{Observaciones IV.} Esta parte funcional continúa la mejora progresiva de las observaciones, centrándose en la calidad final de los registros de residuos. Las observaciones pueden revisarse y corregirse manteniendo reglas de coherencia, rangos científicos y datos estructurados que después serán utilizados en consultas, métricas y exportaciones.
\item \textbf{Taxonomía y catálogo científico.} Esta funcionalidad proporciona las referencias controladas que dan uniformidad a la información científica del sistema. Los catálogos de entornos y tipologías permiten clasificar las sesiones con valores consistentes y reutilizables, mientras que los esquemas compartidos validan taxonomía, coordenadas, cantidades y parámetros de muestreo.
\item \textbf{Interfaz pública y limpieza final.} Esta parte funcional afecta a la presentación general y al cierre del producto. La interfaz pública actúa como primera pantalla de acceso y presentación de la plataforma, mientras que la limpieza final reduce elementos innecesarios y facilita una validación más clara del sistema desarrollado.
\end{itemize}
\subsection{Implementación realizada}

El cuarto sprint tuvo un carácter de integración final y refinamiento funcional.
La implementación se centró en completar la configuración web, reforzar reglas
de permisos, ampliar la administración, permitir edición posterior desde móvil,
consolidar el catálogo científico y preparar el sistema para una presentación y
despliegue más estables.

\paragraph{Backend, permisos y administración.}
En backend se reforzaron los módulos de usuarios, grupos, campañas, catálogo,
sesiones y observaciones. En usuarios se incorporaron operaciones de
actualización de perfil y cambio de contraseña. En grupos se ampliaron las
capacidades de administración con archivado, restauración, invitaciones,
gestión de membresías y generación de copias de seguridad. También se ajustaron
reglas de alcance para diferenciar el acceso de superadministradores,
administradores de grupo e investigadores.

Las campañas y métricas pasaron a respetar de forma más precisa el contexto del
usuario. Cuando no existe grupo activo, el superadministrador puede operar sobre
vistas agregadas, mientras que el investigador mantiene un alcance restringido a
sus campañas, sesiones u observaciones. Estas reglas completaron la separación
de responsabilidades iniciada en los sprints anteriores.

La copia de seguridad administrativa también quedó implementada como una
exportación JSON distinta de la analítica científica. Para un grupo concreto
incluye campañas, sesiones, observaciones, ítems, fotografías, miembros e
invitaciones, junto con metadatos de exportación y un resumen de totales. El
superadministrador dispone además de una copia global de todos los grupos
activos, útil como respaldo operativo del estado de la plataforma.

\paragraph{Edición móvil y sincronización de cambios.}
La aplicación móvil incorporó edición local de sesiones y observaciones ya
creadas. Los repositorios locales permitieron actualizar campos editables y
coordinar esos cambios con la cola de sincronización. Cuando un registro aún
tenía una creación pendiente, las modificaciones se integraron sobre el estado
local antes de enviarlo, evitando duplicar operaciones innecesarias. En backend,
los servicios de sesiones y observaciones añadieron operaciones de actualización
posterior sujetas a permisos, estado y titularidad.

En el caso de las sesiones, la edición posterior distingue entre campos de
descripción y campos que afectan al muestreo. Si la sesión no está finalizada,
pueden actualizarse método, tamaño de cuadrícula, anchura de transecto, entorno
o tipología, y el backend recalcula el área muestreada a partir del recorrido ya
almacenado. En observaciones, la actualización se limita a taxonomía,
coordenadas, fecha, notas e ítems; las fotografías se gestionan mediante sus
endpoints dedicados para mantener separado el ciclo de vida de archivos y datos
estructurados.

\paragraph{Catálogo científico y configuración web.}
El catálogo de entornos y tipologías se gestionó desde un módulo específico de
backend y se consumió desde web y móvil. En el modelo consolidado, la tipología
y el entorno se vinculan a la sesión de muestreo, lo que permite clasificar de
forma más consistente el contexto real de la salida de campo. Los esquemas
compartidos mantuvieron validaciones sobre método de muestreo y parámetros
específicos, como tamaño de cuadrícula o anchura de transecto.

En el portal web se añadió la página de configuración de cuenta, con actualización
de perfil y cambio de contraseña. Además, se consolidaron vistas de
administración de grupos y usuarios, ajustes responsive, traducciones y mejoras
de presentación en páginas principales como dashboard, campañas, observaciones,
mapa y analítica.

\paragraph{Interfaz pública y despliegue.}
La página pública del portal se rediseñó como pantalla de presentación de la
plataforma. En paralelo, se documentó la configuración de despliegue mediante
Docker \cite{docker}, variables de entorno de producción, PostgreSQL \cite{postgresql},
API, web estática y build móvil con EAS \cite{expo-eas}. La guía de despliegue también deja constancia de que el
almacenamiento local de fotografías requiere almacenamiento persistente o un
servicio externo en un entorno productivo como Render \cite{render}.

\subsection{Resultado del sprint}

El resultado del cuarto sprint fue la consolidación final de OmniLitter como una versión integrada, estable y suficientemente completa para ser evaluada dentro del marco del Trabajo de Fin de Grado. Tras haber construido en los sprints anteriores la base funcional, la captura avanzada de datos, la persistencia local, la sincronización, la consulta web, la visualización geográfica y la exportación de información, esta última iteración se orientó principalmente al cierre del sistema, al refinamiento de funcionalidades existentes y a la preparación de una versión más presentable y coherente.

Durante este sprint se reforzaron especialmente las capacidades de administración del portal web, la gestión de usuarios, grupos y permisos, así como los ajustes relacionados con la cuenta de usuario. Estas mejoras permitieron completar aspectos necesarios para que el sistema no solo funcionase desde el punto de vista técnico, sino que también ofreciera una experiencia más ordenada y consistente para los distintos perfiles de usuario. La administración web adquirió así un papel más completo dentro de la plataforma, facilitando la supervisión y organización de la información gestionada por OmniLitter.

Otro resultado relevante fue la consolidación del catálogo científico utilizado para clasificar las observaciones. Esta parte del sistema resultaba especialmente importante, ya que OmniLitter no se limita a registrar información genérica, sino que pretende estructurar los datos de residuos de forma útil para su análisis posterior. La integración del catálogo permitió reforzar la coherencia de los datos recogidos y aportar mayor valor académico y científico a la solución desarrollada. De este modo, las observaciones registradas quedaron mejor alineadas con el objetivo del proyecto: facilitar la monitorización y el estudio de residuos en distintos entornos.

El sprint también permitió mejorar la edición y revisión posterior de los datos. Esta funcionalidad resulta importante porque, en un contexto real de trabajo de campo, la información capturada puede requerir correcciones, ampliaciones o validaciones posteriores. Al incorporar mecanismos de revisión y ajuste, el sistema se hizo más flexible y se aproximó mejor a las necesidades reales de los usuarios, evitando que el flujo de trabajo terminase únicamente en el momento de la captura inicial desde la aplicación móvil.

Desde el punto de vista de integración, el cuarto sprint permitió comprobar que los principales módulos desarrollados a lo largo del proyecto funcionaban de manera coordinada dentro de una misma solución. La aplicación móvil, el backend y el portal web quedaron conectados en un flujo completo que abarca la captura de observaciones en campo, la persistencia local, la sincronización con el servidor, la consulta desde la web, la representación en mapas, la obtención de métricas, la exportación de datos y la administración del sistema. Este resultado es especialmente relevante, ya que evidencia que el proyecto no se compone de funcionalidades aisladas, sino de una plataforma estructurada en torno a un proceso de uso común.

Además, este sprint tuvo un papel fundamental en la estabilización del sistema. Las pruebas regresivas y los recorridos manuales finales permitieron verificar que las modificaciones introducidas durante el cierre no afectaban negativamente a las funcionalidades principales implementadas en iteraciones anteriores. En particular, se revisaron los flujos más representativos del proyecto, prestando atención a la autenticación, la gestión de grupos, la creación y consulta de campañas y sesiones, el registro de observaciones, la sincronización, la visualización web y la exportación de datos. Esta validación final permitió aumentar la confianza en la consistencia del sistema desarrollado.

También se preparó la documentación necesaria para el despliegue y uso de la plataforma, incluyendo indicaciones sobre configuración, variables de entorno, construcción de las aplicaciones y puesta en marcha de los distintos componentes. Este aspecto resultó relevante para cerrar el proyecto desde una perspectiva académica, ya que permite justificar no solo la implementación de la solución, sino también su capacidad para ser instalada, ejecutada y evaluada en un entorno controlado. La documentación de despliegue contribuyó, por tanto, a hacer el sistema más reproducible y comprensible para terceros.

Las incidencias detectadas durante el sprint fueron reducidas y se concentraron principalmente en ajustes derivados de refactorizaciones, adaptación de pruebas y correcciones menores de coherencia visual o funcional. Estas incidencias no alteraron el alcance principal del proyecto, sino que formaron parte del proceso normal de estabilización de una versión final. Su impacto fue limitado y permitió cerrar el desarrollo con una base técnica más ordenada y preparada para posibles ampliaciones futuras.

El grado de cumplimiento del sprint se considera alto, ya que se alcanzó el objetivo principal de transformar el prototipo funcional construido durante las iteraciones anteriores en una versión final más integrada, documentada y apta para evaluación. El sprint no se centró únicamente en añadir nuevas funcionalidades, sino en asegurar que las ya desarrolladas funcionasen correctamente dentro de un mismo flujo, reforzando la calidad global del sistema y su presentación como producto software.

En consecuencia, el cuarto sprint supuso el cierre natural del proceso de desarrollo incremental de OmniLitter. Su principal aportación fue consolidar el sistema, reducir inconsistencias, completar funcionalidades de administración y revisión, preparar la documentación de despliegue y validar los recorridos principales de uso. Como resultado, el proyecto alcanzó un estado suficientemente maduro para demostrar los objetivos planteados en el Trabajo de Fin de Grado, dejando además identificadas sus limitaciones y una base técnica preparada para futuras líneas de mejora.

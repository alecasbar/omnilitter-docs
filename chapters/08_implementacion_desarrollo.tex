La validación del proyecto se organizó de forma incremental, siguiendo la misma división por sprints utilizada durante el desarrollo. El objetivo fue comprobar, en cada etapa, que las funcionalidades incorporadas no solo estaban disponibles desde la interfaz, sino que respetaban las reglas de negocio, los permisos, los contratos de datos y las restricciones definidas para el sistema.

La estrategia de pruebas combinó dos enfoques complementarios. Por un lado, se implementaron pruebas automatizadas sobre las partes más críticas del proyecto: esquemas compartidos, servicios de la API y determinadas utilidades del portal web. Por otro lado, los recorridos completos de usuario, especialmente aquellos relacionados con la aplicación móvil, el uso offline, la cámara, la ubicación y la sincronización, se validaron mediante pruebas manuales funcionales descritas como procedimientos de ejecución.

Esta separación permite reflejar con precisión el alcance real de la validación. La memoria no atribuye al proyecto pruebas automatizadas que no existen, pero sí deja constancia de que las funcionalidades principales fueron comprobadas mediante el tipo de prueba más adecuado a su naturaleza.

Esta decisión permitió comprobar reglas de negocio relevantes sin depender de una base de datos PostgreSQL real ni de la ejecución completa de rutas HTTP.

\manualCaseCell{CU-01}{Registrarse, autenticarse y gestionar la sesión.} &
\textbf{PM-CU01-01 -- Registro de nuevo usuario}
\begin{itemize}[leftmargin=1.2em,itemsep=0.15em,topsep=0.2em]
\item Abrimos el formulario de registro del portal web.
\item Enviamos datos incompletos, correo inválido y contraseña débil, y vimos que el sistema mostraba errores de validación.
\item Registramos una cuenta con nombre, correo único y contraseña válida.
\item Comprobamos que el usuario accedía correctamente a la zona autenticada.
\end{itemize}
\textbf{PM-CU01-02 -- Inicio y cierre de sesión en web y móvil}
\begin{itemize}[leftmargin=1.2em,itemsep=0.15em,topsep=0.2em]
\item Intentamos iniciar sesión con credenciales incorrectas y vimos que se rechazaban.
\item Iniciamos sesión con credenciales válidas y accedimos a pantallas privadas.
\item Recargamos la web y reabrimos la aplicación para comprobar que la sesión se mantenía.
\item Cerramos sesión y verificamos que ya no era posible acceder a zonas protegidas.
\end{itemize}
\textbf{PM-CU01-03 -- Protección de rutas privadas}
\begin{itemize}[leftmargin=1.2em,itemsep=0.15em,topsep=0.2em]
\item Abrimos el portal sin sesión activa.
\item Accedimos manualmente a rutas privadas como dashboard, campañas, mapa o configuración.
\item Comprobamos que todas redirigían al acceso público.
\item Iniciamos sesión y vimos que las mismas rutas mostraban los módulos permitidos.
\end{itemize}
\textbf{PM-CU01-04 -- Cambio de contraseña del usuario autenticado en portal web}
\begin{itemize}[leftmargin=1.2em,itemsep=0.15em,topsep=0.2em]
\item Iniciamos sesión con un usuario existente.
\item Intentamos cambiar la contraseña con contraseña actual incorrecta.
\item Intentamos reutilizar la contraseña vigente como nueva contraseña.
\item Introdujimos contraseña actual correcta y nueva contraseña válida.
\item Cerramos sesión e iniciamos sesión con la nueva contraseña correctamente.
\end{itemize}

\textbf{PM-CU03-01 -- Creación y edición de grupo}

\textbf{PM-CU03-02 -- Gestión de miembros y roles de grupo}
\begin{itemize}[leftmargin=1.2em,itemsep=0.15em,topsep=0.2em]
\item Añadimos un usuario existente como miembro de un grupo, desde una cuenta de superadministrador.
\item Cambiamos su rol y vimos que las acciones disponibles se ajustaban al nuevo permiso.
\item Expulsamos al miembro del grupo.
\item Y comprobamos que ya no existía en el grupo.
\item Ingresamos como un usuario normal para gestionar un grupo del que somos administradores.
\item Dimos permisos de administrador a un usuario del grupo.
\item Comprobamos que no se le puede cambiar el rol a un administrador de grupo.
\end{itemize}
\textbf{PM-CU03-03 -- Invitación y aceptación desde la app móvil}
\begin{itemize}[leftmargin=1.2em,itemsep=0.15em,topsep=0.2em]

\item Iniciamos sesión con el usuario invitado.
\item Nos dirigimos a las invitaciones pendientes y aceptamos la invitación.
\item Comprobamos que el nuevo grupo quedaba disponible para trabajar.
\item Del mismo modo lo probamos en la aplicación móvil.
\end{itemize}

\item Archivamos el grupo y comprobamos que sus campañas activas quedaban cerradas o no disponibles.
\item Restauramos el grupo.
\item Intentamos eliminar definitivamente un grupo activo y vimos que el sistema respetaba permisos y estados.
\end{itemize}
